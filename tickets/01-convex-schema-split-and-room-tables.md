# Ticket 1: Convex Schema Split and New Room Tables

## Objective
Decouple Better Auth generated schema from app-specific schema so we can safely add poker room tables without losing custom changes on auth schema regeneration.

## Why This Ticket Exists
`convex/schema.ts` is currently fully auto-generated by Better Auth. Any direct edits are overwritten by `@better-auth/cli generate`. We need a stable split:
- `convex/schema.auth.ts`: generated auth tables only
- `convex/schema.ts`: manual composition of auth tables + app tables

This is a prerequisite for all room/presence/chat work.

## Current State
- `convex/schema.ts` is generated and contains auth tables: `user`, `session`, `account`, `verification`, `jwks`.
- Better Auth integration depends on these table names and indexes.
- No room/presence/chat tables exist.

## Scope
- Create app-owned table definitions for `rooms`, `players`, `messages`.
- Preserve all existing Better Auth tables and indexes unchanged.
- Make schema regeneration safe (auth regen does not wipe app tables).

## Out of Scope
- Convex mutations/queries for room lifecycle
- Lobby or room UI
- Cleanup cron job

## Implementation Plan

### 1) Split generated and manual schema files
- Keep Better Auth generated output in `convex/schema.auth.ts`.
- Create manual `convex/schema.ts` that:
  - imports `defineSchema`, `defineTable`, `v`
  - imports auth table map from `schema.auth.ts`
  - defines app tables (`rooms`, `players`, `messages`)
  - exports merged schema via `defineSchema({ ...authTables, ...appTables })`

### 2) Define app table shapes
Use these exact fields and enum values.

#### `rooms`
- `code: v.string()`
- `status: v.union(v.literal("open"), v.literal("closed"))`
- `maxPlayers: v.number()`
- `hostPlayerId: v.id("players")`
- `createdAt: v.number()`
- `lastActiveAt: v.number()`

Indexes:
- `.index("code", ["code"])`
- `.index("status_lastActiveAt", ["status", "lastActiveAt"])`

#### `players`
- `roomId: v.id("rooms")`
- `name: v.string()`
- `seatIndex: v.number()`
- `playerToken: v.string()`
- `isHost: v.boolean()`
- `status: v.union(v.literal("active"), v.literal("left"))`
- `lastSeenAt: v.number()`

Indexes:
- `.index("roomId", ["roomId"])`
- `.index("roomId_status", ["roomId", "status"])`
- `.index("roomId_seatIndex", ["roomId", "seatIndex"])`
- `.index("playerToken", ["playerToken"])`

#### `messages`
- `roomId: v.id("rooms")`
- `playerId: v.id("players")`
- `text: v.string()`
- `createdAt: v.number()`

Indexes:
- `.index("roomId_createdAt", ["roomId", "createdAt"])`
- `.index("playerId", ["playerId"])`

### 3) Update generation workflow
Set Better Auth schema generation to target `convex/schema.auth.ts`.

Expected command pattern:
```bash
npx @better-auth/cli generate --output ./convex/schema.auth.ts -y
```

Also update any comments/docs/scripts referencing `convex/schema.ts` as generator output.

### 4) Regenerate Convex types
Run Convex codegen after schema update so app code can reference new table IDs and validators.

## Data and Validation Constraints
- `rooms.code` must be unique in app logic (enforced in mutation layer later).
- `players.seatIndex` range is `0..maxPlayers-1` (enforced in mutation layer later).
- `messages.text` max length will be enforced in mutation layer later.
- This ticket defines schema-level type constraints only.

## File-Level Changes
- Create: `convex/schema.auth.ts` (generated file)
- Replace: `convex/schema.ts` (manual merged schema)
- Update comments/docs where generator output path is documented (if present)

## Acceptance Criteria
1. `convex/schema.ts` is manual and no longer marked as generated.
2. Better Auth tables exist unchanged with original names and indexes.
3. New tables `rooms`, `players`, `messages` are present with the specified fields/indexes.
4. Convex codegen completes and exposes new table types in `convex/_generated/dataModel.d.ts`.
5. Running Better Auth schema generation does not remove app tables.

## Verification Checklist
- Regenerate auth schema to `schema.auth.ts` and confirm file updates.
- Run Convex codegen and verify no schema errors.
- Inspect generated types for new table IDs.
- Spot-check Better Auth flow still reads/writes auth tables.

## Risks
- Breaking Better Auth by renaming or altering auth table definitions.
- Circular table ID references (`rooms.hostPlayerId -> players`, `players.roomId -> rooms`) causing schema issues if typed incorrectly.
- Forgetting to update generator output path leads to accidental overwrite.

## Dependencies
- None. This is the foundational schema ticket for all subsequent room/presence/chat work.

## Done Definition
This ticket is done when schema split is in place, app tables are defined and indexed, Convex types regenerate successfully, and auth schema regeneration is safe and repeatable.
