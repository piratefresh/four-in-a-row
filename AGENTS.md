# Repository Instructions

## Tooling
- Use **Bun** for package management and scripts: `bun install`, `bun run dev`, `bun run build`, `bun run test`, `bunx convex dev`. Do not use `npm`, `npx`, or `yarn` even when README/package scripts mention them.
- `README.md` still contains scaffolded npm commands; trust `package.json`, `bun.lock`, and this file instead.

## Common Commands
- Frontend dev server: `bun run dev` (Vite/TanStack Start on port 3000).
- Convex dev/codegen server: `bunx convex dev` in a separate terminal; required for generated Convex types/API and local backend behavior.
- Unit tests: `bun run test`; focused test: `bun run test convex/gameRules.test.ts` or any specific `*.test.ts` path.
- Build: `bun run build`.
- E2E: `bun run test:e2e`; Playwright starts the frontend but tests expect a real Convex deployment at `CONVEX_URL` or `http://127.0.0.1:3210`.
- CSW word index generation: `bun run generate:csw24-index` updates generated dictionary/index data.

## Environment
- Frontend startup requires `VITE_CONVEX_URL`; `src/router.tsx` throws if missing.
- Convex/OpenRouter AI paths use `OPENROUTER_API_KEY`, optional `OPENROUTER_MODEL`, optional `OPENROUTER_BASE_URL`, and `BETTER_AUTH_URL`/`BETTER_AUTH_BASE_URL` for auth/request origin.
- OpenRouter is the only configured AI provider in current source; ignore stale docs mentioning NVIDIA NIM.

## Generated Files
- `src/routeTree.gen.ts` is generated by TanStack Router from `src/routes`; do not hand-edit unless unavoidable.
- `convex/_generated/*` is generated by Convex; regenerate via `bunx convex dev` after schema/function signature changes.
- `convex/csw24.generated.ts` is generated dictionary data; use `bun run generate:csw24-index` rather than manual edits.

## Architecture Notes
- Frontend routes live in `src/routes`; keep route files focused on loading/navigation/orchestration and move rendering-heavy UI to `src/components/...`.
- Convex public API aggregators are root files like `convex/games.ts` and `convex/rooms.ts`; many implementations live in `convex/games/*` and `convex/rooms/*`.
- Auth is Better Auth through Convex (`convex/auth.ts`, `src/lib/auth-*`, `ConvexBetterAuthProvider` in `src/routes/__root.tsx`).
- Game constants and validators are in `convex/gameState.ts`; avoid duplicating betting constants like ante/raise ladder.
- AI betting/showdown code is centered in `convex/ai.ts`, `convex/aiStrategy.ts`, `convex/aiPrompts.ts`, and game callers in `convex/games/gamesBetting.ts` / `gamesShowdown.ts`.

## Component Conventions
- For React/Next-style performance, data-fetching, or component refactors, consult the `vercel-react-best-practices` skill.
- For component API/composition work such as compound components, render props, context boundaries, or reducing boolean props, consult the `vercel-composition-patterns` skill.
- Prefer shared context or focused hooks over long prop chains for room/game state.
- Keep component prop surfaces small and intentional; extract repeated or deeply nested JSX into named components.
- Keep components near `<= 250` lines when practical; move large showdown/results/UI sections into `src/components/...`.
